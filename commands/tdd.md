# TDD（テスト駆動開発）ワークフロー

Agent Teamsフローに統合されたTDD手順。実装メンバーが使う。

## TDDサイクル

```
RED → GREEN → REFACTOR → REPEAT

RED:      失敗するテストを書く
GREEN:    テストを通す最小限のコードを書く
REFACTOR: テストが通ったままコードを改善する
REPEAT:   次の機能/シナリオへ
```

## 手順

### 1. 設計書を全文読む
設計書の入出力仕様・制約を正確に把握する。要約しない。

### 2. テストを先に書く（RED）
- 設計書の仕様に基づいてテストを作成する
- ハッピーパスだけでなくエッジケースも含める：
  - None/空入力
  - 境界値（最小/最大）
  - 異常入力（不正な型、範囲外の値）
  - 安全弁の動作（上限到達時、例外発生時）
  - エラーパス（ファイル不在、接続失敗等）

### 3. テストを実行して失敗を確認
```bash
python -m pytest path/to/test_file.py -v
```
テストが「正しい理由で」失敗することを確認する（NotImplementedError等）。

### 4. 最小限の実装を書く（GREEN）
テストを通すためだけのコードを書く。余計な機能を追加しない。

### 5. テストを実行して成功を確認
```bash
python -m pytest path/to/test_file.py -v
```
全テストがパスすることを確認する。

### 6. リファクタリング（REFACTOR）
テストが通ったまま、コードの品質を改善する：
- 重複を除去
- 命名を改善
- 複雑な関数を分割
- リファクタ後も全テストがパスすることを確認

### 7. 実際に起動して確認
ユニットテストだけでなく、実際にBashでコマンドとして起動して動作を確認する。
テストが通ることと実際に使えることは別。

## Agent Teamsフローでの位置づけ

```
設計メンバー → 設計書作成
    ↓
解析メンバー → 設計のリスク確認
    ↓
実装メンバー → TDDで実装（このワークフロー）
    ↓
解析メンバー → 実装のリスク再確認
    ↓
レビューメンバー → コード品質検査
```

## テストのアンチパターン（避けるべきこと）
- 実装の内部状態をテストする（振る舞いをテストすべき）
- テスト間で状態を共有する
- アサーションが甘い（何も検証していないテスト）
- 外部依存をモックしない（DB、API、ファイルI/O）

## 禁止事項
- テストを書く前に実装を書くこと
- テストの失敗を確認せずに実装に進むこと
- テストが通らない状態でリファクタリングすること
- 設計書にない機能を勝手に追加すること
