# 競合仮説デバッグ

bugfix時に根本原因の候補が2つ以上ある場合、候補ごとに別サブエージェントを並列起動して調査する。

## 前提条件
- バグの症状が明確であること
- 根本原因候補が2つ以上特定されていること
- `/bugfix` フローのステップ2（根本原因の調査）で使用

## 手順

### 1. 仮説のリスト化
根本原因の候補を明示的にリスト化する（最大3つ）。

```
仮説A: [具体的な原因の説明]
仮説B: [具体的な原因の説明]
仮説C: [具体的な原因の説明]（任意）
```

3つ超なら先にthinkerで絞り込む。

### 2. 並列調査の起動
各候補に対してimplementerを並列起動する。

**各エージェントへのプロンプト:**
```
CLAUDE.mdを全文読んでください。

## バグ調査タスク
症状: [バグの症状]
担当仮説: [仮説X]
調査対象ファイル: [関連ファイルリスト]

## 調査手順
1. 仮説が正しいと仮定して、原因箇所を特定する
2. 再現テストを書く（仮説が正しければ失敗するテスト）
3. テストを実行して結果を報告する
4. 仮説を支持する証拠と反証を列挙する

## 出力形式
- 仮説の支持度: HIGH / MEDIUM / LOW
- 証拠リスト
- 反証リスト
- 再現テストの結果
- 修正案（仮説が正しい場合）
```

### 3. 結果の比較
各サブエージェントの結果を比較し、最も証拠が多い候補を採用する。

判定基準:
- 再現テストが失敗する（仮説を支持する）候補を優先
- 証拠の具体性と数で判断
- 同等なら thinker で追加分析

### 4. 記録
- 採用候補: 修正を実施
- 不採用候補: `bug_investigation_log.md` に記録（将来の類似バグで再検討可能）
- 教訓があれば `lessons_registry` に追加

## 安全弁
- 候補は最大3つまで
- 各調査エージェントのタイムアウト: 10分
- 3候補全て LOW なら thinker に差し戻し
