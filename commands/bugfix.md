# バグ修正フロー

バグ・不具合・エラー発生時の手順。プロセスは作業の一部。
各フェーズ移行時に `memory_search` でコンテキスト依存の記憶検索を実行する（Attention Residual: 現在の入力に応じた動的重み付けで全記憶にアクセス）。

## 手順

### 1. コンテキスト依存の記憶検索（Attention Residual）
```
memory_search query="[バグの症状・箇所のキーワード]" で類似バグ・過去の修正パターンを検索
→ 現在のコンテキスト（何を修正しようとしているか）に基づく動的検索
→ 見つかった教訓をSTMに書く（以降のステップへの残差接続）
```
静的リストを読むのではなく、**今の状況に関連する**記憶を動的に引き出す。

### 2. 根本原因を調査する
表面的な修正で終わらせない。「なぜ」を問い続けて根本まで掘る。

### 3. TDDで修正する（/tdd参照）
```
RED:      バグを再現するテストを書く
GREEN:    テストを通す最小限の修正を書く
REFACTOR: テストが通ったまま改善
```
テストを書く前に実装を書くことは禁止。

### 4. 実際に動かして確認する
テスト通過だけで終わらない。実際にコマンドとして起動して動作確認。

### 5. 類似パターンを全箇所grepして修正する
```
同じバグが他の箇所にもないかgrepで検索。1箇所だけ直して終わりにしない。
見つかったら全箇所を修正する。2箇所以上あれば共通化（定数/ユーティリティ）する。
```

### 6. 再発防止の構造を作る（必須。省略禁止）
```
根本原因 = バグ。バグの修正だけでは50%。再発防止の構造を作って100%。
- 定数化（マジックナンバー散在 → 共有定数）
- ユーティリティ関数（同じ処理の重複 → 共通関数）
- behavior-guardルール（プロセスの問題 → 検出+ブロック）
- テスト（設計制約 → テストで強制）
- lint/grep検出（廃止パターン → 自動検出）
「次から気をつける」は対策ではない。構造を作れ。
「低リスクだから不要」はごまかし。根本原因があるならバグ。直せ。
```

### 7. bug_investigation_log.md に記録する
```
- 発生箇所
- 症状
- 根本原因
- 修正内容
- テスト
- 再発防止の構造（何を作ったか具体的に）
```

### 8. 必要なら教訓レジストリに追加する
根本原因がプロセスの問題（手順省略、確認不足等）なら教訓として記録。
