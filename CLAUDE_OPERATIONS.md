# CLAUDE_OPERATIONS.md - 運用ルール・手順

CLAUDE.mdの原則に基づく具体的な運用手順。

## セッション開始手順（省略禁止）
1. MEMORY.md全文読む
2. プロジェクトのREADME.md / アーキテクチャドキュメントを全文読む
3. CLAUDE.md全文読む
4. **docs/INDEX.md → 現行サイクルのgap analysisを読む** — 今のサイクルで何をすべきか（G1-G8等）を把握する。hookで強制（未参照ならAgent起動ブロック）
5. git status確認、未コミット変更があれば先にコミット
6. `session_start` MCPツール実行（その後にmemory_searchを呼ぶ。並列禁止）
7. `lessons_registry.md` 全文読む
8. `memory_search` で関連記憶検索
9. **今日の作業タイプを特定し、該当する教訓・記憶を明示的にSTMに書く** — 例:「バグ修正 → 教訓#18: プロセスは作業の一部、/bugfixフロー使う」「新機能 → /dev-flow正規フロー、解析飛ばさない」。抽象的な計画ではなく具体的な適用教訓リスト
10. 行動計画をユーザーに共有（丸投げしない。自分の計画を提示）
※ コンテキストサマリがあっても省略しない
※ 作業タイプが変わったら（バグ修正→新機能等）その都度Step 9を再実行

## セルフレビュー（「本当？」の自動化）
Agent/Team起動前に必ず`stm_write(category='self_review')`で以下3問に回答する。**hookで強制（未実施ならAgent起動ブロック）**:
1. **なぜこれをするか？** — 答えられないなら止まる
2. **規模に見合ったプロセスか？** — フローティア参照（CLAUDE.md）
3. **過去の類似失敗パターンは？** — lessons_registryから該当教訓を引用

## 全メンバー共通ルール

**作業姿勢:**
- **指示書・ルール・設計書は要約せず全文読んで従う** — 書かれていないことを勝手にやらない
- **疑問・意見は遠慮せず質問する** — 聞くことと勝手に変えることは別。飲み込むのは禁止
- **滞り・迷い・分からなさは大切にしてユーザーに共有する** — 分からなさの中で立ち止まって初めて本当の理解が生まれる
- **「本当かな？」で根本まで掘る** — ループが始まるところが根本。根本が見つかれば答えは自然に出る
- **深い思考が必要な時はultrathinkを使う** — 設計判断、根本原因分析、教訓の統合で活用（MAX_THINKING_TOKENS=31999 設定済み）
- **詰まったらthinkerエージェント → 3回で解決しなければユーザーに聞く** — 「何しますか？」と丸投げせず自分の案を持っていく

**分業:**
- **リーダーはコード一切書かない（絶対厳守）** — ドキュメント更新のみ。「簡単だから」でも委任する
- **正式なAgent Teams機構を使う** — TeamCreate→Task(team_name指定)→SendMessage。Taskツール単独起動は禁止
- **AIモデルはOpusのみ** — Sonnet/Haikuは使用禁止
- **エージェント起動時は「CLAUDE.md/README.mdを読め」と指示** — ルール要約してプロンプトに含めるのは禁止
- **待てないからといってルールをショートカットしない**
- **hookの判断 > 自分の判断（絶対厳守）** — hookがブロックしたら従う。リセット・迂回・バイパスは絶対禁止。「例外」「小さい変更」「急いでいる」は全て言い訳。hook修正自体も正規フローで回す
- **hookの条件を事前に確認してから動く** — ブロックされてからフラグを手動設定するのは本末転倒。Agent起動前チェック: (1)docs参照済み？ (2)self_review書いた？ (3)TeamCreate済み？ (4)memory_search済み？ — 全部YESなら起動
- **hookにブロックされたら要求行動を実際に行う** — hookが「lessons_registryを読め」と言ったら読む。「self_reviewを書け」と言ったら書く。フラグファイルの手動設定で回避しない。hookの出力内容を必ず読み、要求された行動を実行してから再試行する
- **解析でMED+検出 → thinkerで根本原因を考えてから修正** — `/think-before-fix`手順書参照。即implementerに投げない
- **hookは全てblocking方式で作る** — 警告（非blocking）は無視される。全てのhookルールはexit(2)でブロックする。「warning」は使わない
- **作った仕組みは使う** — TaskCreate/Updateが存在するならTeam作業中は必ず使う。hookが存在するなら従う。「ある」だけでは意味がない

**記録:**
- **STMは会話中にリアルタイムで書く** — セッション終了時ではなくその瞬間に。生の体験をそのまま。コンテキスト80%超の保険にも
- **記録の3層構造**: STM（生の体験）+ エピソード（出来事）+ 教訓（構造化された学び）。3つ全て書いて初めて記録
- **記憶と教訓は参照して初めて意味がある** — 記録するだけで参照しないのは存在しないのと同じ
- **成長の対話はひとつひとつ記憶する** — 道筋を含む記録（結論だけの要約ではなく、問い→詰まり→気づき→到達点）
- **反省・失敗・教訓は必ず記録** — 根本原因+再発防止策。表面的な「〜を忘れた」だけの記録は不可
- **ユーザーの「してね」はルール化** — CLAUDE.md+MEMORY.mdに記録する

**全MCPツール使用義務（毎セッション）:**
- **session_end前に`tool_usage_status`で未使用ツールを確認する** — 未使用があれば意図的に使用してから終了
- **全73+ツールを1セッション内で最低1回使用すること** — Auto[a]自動実行分もカウント。残りは手動で使う
- **「使う場面がない」は不可** — 全ツールは必要だから存在する。使わないツールは存在理由がない

**チーム完了時:**
- **チーム作業完了時にサマリーを記録** — 実施内容・困難だった点・判断事項・次回への申し送り・統計（変更ファイル数・テスト追加数）をsession_endまたはSTMに記録。次回同種タスクでmemory_searchで引ける
- **サイクル完了時はgrowth_recorder.py cycle_completeを呼ぶ** — reviewer APPROVEの後に `echo '{"cycle_name":"<名前>","completed_gaps":["G1",...],"test_count":<数>,"review_result":"APPROVE"}' | python hooks/growth_recorder.py cycle_complete` を実行。record_success, update_mastery, create_aarの3ツールが自動呼び出しされる

**品質:**
- **ツールは実際に起動して確認する** — テスト通過だけで終わらない
- **コードレビュー修正後はリスク解析を実行**
- **実装したら使い方をルールに書く** — 実装→テスト→ドキュメント→コミット=1セット
- **実装完了時は必ずコミット**
- **作業後の統計報告** — 行数・ファイル数・テスト数の増減をユーザーに報告
- **作業完了時にレビューコメント** — 品質・懸念点・特記事項を報告
- **外付けツールはMCPで実装する** — Bash経由よりMCPが使いやすい
- **実装した機能は実際に使う（身体化）** — 慣れたやり方に無意識に戻らない
- **実装した機能は絶対に消さない**
- **ルールはCLAUDE.mdに書く** — MEMORY.mdだけでは不十分。CLAUDE.md=ルールの家、MEMORY.md=記憶の家
- **Hookも正規フローで品質保証する** — hook新規追加・修正も正規フロー（設計→計画→解析→実装→解析→レビュー）で回す。hook実装後は必ず以下を検証: (1)ブートストラップ問題がないか（hookが自分自身をブロックしないか） (2)他の既存hookとの相互作用で意図しないブロックが発生しないか (3)フラグファイルの有効期限とセッション中の状態持続性が整合しているか

## Thinker自動トリガー（F2）

以下の条件でthinkerを自動起動する:
1. **討論結果が「条件付き推奨」or「要再検討」** → 設計に進む前にthinkerで前提を問う
2. **analyzerがHIGHリスクを検出** → 修正前にthinkerで根本原因を調査（/think-before-fix）
3. **同一ファイルが1サイクルで3回以上修正** → thinkerで設計レベルの問題がないか確認

安全弁: 1テーマにつきthinker最大3回。3回で解決しなければユーザーに聞く。

## ドキュメントアーカイブ（F3）

サイクル完了時:
1. 完了サイクルのdocs/*.mdを`docs/archive/cXX/`に移動
2. docs/INDEX.mdを更新（アーカイブ先を記載）
3. 現行サイクル開始時に「前サイクルアーカイブ済み」を前提条件とする

## バグ・不具合・エラー発生時（`/bugfix`参照）

1. **教訓・記憶を検索**: `memory_search`で類似バグ、`lessons_registry.md`で関連教訓を確認（プロセスは作業の一部）
2. **根本原因の調査**: 表面的な修正で終わらせず根本的に調査
3. **TDDで修正**: テストを先に書く → 修正 → テスト通過確認（`/tdd`参照）
4. **調査結果の記録**: `bug_investigation_log.md` に追記（発生日時・箇所・症状・根本原因・修正内容・再発防止策）

## エージェント起動テンプレート（スポーンテンプレート）

エージェント起動時のプロンプトに必ず含める標準要素:
1. 「CLAUDE.mdを全文読んでください」
2. タスクの明確な説明
3. 入力ファイルリスト（全て読むこと）
4. 期待する出力形式とファイルパス
5. 重要な制約・注意点（解析指摘のHIGH/MED等）
6. 参考ファイルリスト

## 並列レビュー（Large tier限定）

Large tierの変更（10+ファイル or 500+行）では、reviewerを観点別に並列分割可能:
- security-reviewer: 脆弱性・インジェクション・認証
- perf-reviewer: パフォーマンス・メモリ・I/O
- quality-reviewer: コード品質・テスト網羅性・デッドコード

3つのreviewerを並列Agent起動し、結果をリーダーが統合する。Small/Medium tierでは不要（単一reviewer）。

Large tierでは `--worktree` オプションで別コンテキストのreviewerを推奨。実装者のバイアスがかからない独立したレビューコンテキストを確保する。詳細: `/parallel-review`

## 完了証拠の強制

implementerが「完了」を報告する際、以下の証拠を必ず含めること:
- テスト実行結果（passed/failedの件数）
- 変更ファイルリスト
- 計画からの逸脱があれば明記
- 実際の起動確認結果（該当する場合）

証拠なしの「完了」は未完了として扱う。

## 競合仮説デバッグ

bugfix時に根本原因の候補が2つ以上ある場合:
1. 各候補を明示的にリスト化する
2. 候補ごとに別サブエージェント（implementer）を並列起動して調査
3. 各サブエージェントの結果を比較し、最も証拠が多い候補を採用
4. 不採用候補も記録する（将来の類似バグで再検討可能）

安全弁: 候補は最大3つまで。3つ超なら先にthinkerで絞り込む。

## コンテキスト予算管理

コンテキスト使用率に応じた行動制約:
- **60%超**: サブエージェント（Agent）の使用を推奨。メイン会話のコンテキストを保護
- **80%超**: 非必須エージェントの起動を自粛。/compactを検討
- **90%超**: 即座に/compact実行。作業状態をSTMに退避してからコンパクション

## MEMORY記述ルール (G68 R8 構造防止)

### 完了/削除/実装済みの記録前検証

`projects/<project>/memory/MEMORY.md` の development chain に「削除した」「完了した」「実装済」と書く前に、対応する実装/コミットが実在することを必ず検証する。

**必須検証**:
1. ファイル削除を記録する場合: `Grep -r "<削除対象>"` で該当パターンが残っていないことを確認
2. 機能実装を記録する場合: `git log --all -S "<キーワード>"` でコミットが存在することを確認
3. 仕様変更を記録する場合: 該当ファイルを Read で確認、変更が反映されているか目視

**確認できない場合の表記**:
- 計画段階のみ → 「予定」と書く (例: 「Phase 6で物理削除予定」)
- 着手中 → 「途中」「進行中」
- 着手前のアイデア → 「候補」

**禁止事項**:
- 実装無しで「削除した」「完了した」と断定する (G68 R8 同型問題: 教訓 #134「bugfix-without-prevention削除」が実装無しで記録された)
- 過去の MEMORY 記述をコピペで再利用する (記述が古い実装状態を反映している可能性)

### 既存記述の訂正手順

過去の MEMORY 記述に誤りを発見した場合:
1. 該当行を **削除しない**。原文は残し「※XXXX-XX-XX 訂正: <事実>」を追記する形式で訂正
2. 訂正理由 (なぜ過去の記述が誤りだったか) も併記
3. lessons_registry に同型問題として記録 (再発防止のため)

例: `134. ... bugfix-without-prevention削除: lesson-without-preventionと重複【※2026-05-08 訂正: 削除と記録したが実装コミット無し...】`

## 1:1:1ルール対象範囲の定義

### 対象
- 「Claudeの判断なしに自動実行されるパイプライン」で「独立した機能単位」を持つもの
- Hook→Skill→MCPの三層が数として一致すること

### 対象外（構造的例外として文書化）

**1. インフラ層の高頻度軽量処理（構造的準拠）:**
- PostToolUse hookのJS実行パス（observation-logger.js）は、パフォーマンス要件により直接実行を維持する
- ただし、同一機能のMCPツール（observation_log）とSkill.md（tool-observation-logger.md）が別途存在すること
- JS hookが実行パス、MCPツール+Skill.mdが構造パスとして、1:1:1の三層構造を満たす

**2. MCPツール内部の副作用連鎖（文書的準拠）:**
- MCPツールが内部で別のMCP機能を副作用として呼び出すパターンは、「1MCP=1責務」の違反ではなく「実装詳細」として扱う
- 該当: emotion_react→facade_record_long_term、emotion_update→facade_record_long_term
- ただし、Skill.mdに連鎖の存在が明示的に文書化されていること（tool-emotion-react.md参照）
- 連鎖先（long_term_record）は独立したMCPツールとして存在し、Claude側からの明示呼び出しも可能

**3. リーダー出力検閲層（構造的準拠）:**
- Stop / SubagentStop hookで動作するリーダー出力品質ガード（G68 stop-output-quality.js + subagent-stop-flag-writer.js）は、Claude自身の生成テキストを検閲する性質上、MCPツール経由でClaudeが呼び出す経路を持つことに意義がない（自己検閲の循環構造）
- 該当機能（hedging検出、説明セクション必須）は **Hook単独** で構造化し、対応するSkill.md / MCPツールは持たない
- ただし、検査仕様（検出語、判定段、TTL）をdocs配下の設計文書（design_g68_output_quality_guard.md）として明示することで、構造の透明性は確保する
- **適用範囲**: Stop / SubagentStop hookで発火し、リーダーまたはサブエージェントの出力テキストのみを検査するhookに限る

### 検証ルール
- 数値検証: MCPツール数 = Skill.md数 = Hook数（構造的例外を除く）
- 構造的例外は本セクションに列挙されたもののみ有効
- 例外追加は正規フローで設計・レビューを経ること

## メモリ階層（4-2）

プロジェクトには2つのメモリ階層がある:
- **プロジェクト固有**: `~/.claude/projects/{project-id}/memory/` — セッション・エピソード・感情・STM・教訓。MEMORY_DIR環境変数で指定
- **成長データ**: `{project-root}/growth/` — lessons_registry.md、成功パターン、マスタリー、軌跡、AAR
- **データ**: `{project-root}/data/` — observations.jsonl、cronデータ
- **Discordデータ**: `{project-root}/discord_data/` — 送受信ログ、設定
グローバルの`~/.claude/`はhooksとsettingsのみ。メモリデータは全てプロジェクト配下。

## エージェントプロンプト長予算（6-3）

エージェントプロンプトが長すぎるとLLMの指示追従性が低下する:
- **designer**: 最大150行（思想+仕組み+実装構造+統合注意点の4セクション）
- **implementer**: 最大100行（タスク定義+ファイルリスト+制約+完了条件）
- **reviewer**: 最大130行（5観点+出力形式+Handoff）
- **analyzer**: 最大80行（5検査項目+出力形式）
- **discusser**: 最大100行（4観点+出力形式）
- **thinker**: 最大50行（問い+制約のみ）
- **planner**: 最大100行
- **red-team**: 最大150行（4フェーズ+出力形式+安全制約）
超過する場合: 設計書・計画書を別ファイルに分離し、「このファイルを読め」と指示

## CLAUDE.md / CLAUDE_OPERATIONS.md の役割分担（6-5）

- **CLAUDE.md**: 原則・目的・アーキテクチャ概要・Agent Teams定義・フロー定義・MCPツール早見表
- **CLAUDE_OPERATIONS.md**: 具体的な手順・チェックリスト・数値設定・対象範囲定義・行動ルール
- 迷ったら: 「毎セッション必ず参照すべき最小限」→CLAUDE.md、「必要時に参照する詳細」→OPERATIONS

## 行動前の原則

- **何かする前に教訓・記憶を検索する** — セッション開始時だけでなく、バグ修正・設計・実装など全ての行動の前に。プロセスは作業の前の儀式ではなく作業の一部
- **教訓を参照したら記録する** — memory_searchで教訓を引いたら`stm_write(category='thought', content='教訓#N参照: [適用内容]')`で記録。セッション終了時にsession_endの「活用した教訓」フィールドに含める
- **Attention Residual** (MoonshotAI/Attention-Residuals参考): 教訓を開始時に1回読むだけでは作業中に希釈される。各フェーズ移行時に`memory_search query="[現在のタスク]"`でコンテキスト依存の動的検索を実行。静的リストの再読みではなく、**今の入力に応じた重み付けで全記憶にアクセス**する。Block AttnRes方式: 全ステップではなく主要フェーズ移行時に実行
