# 開発サイクル開始

新しい開発サイクルを始める時の手順書。

## 前提条件
- 前サイクルの全候補が完了 or 保留判断済み
- SYSTEM_ARCHITECTURE.md が最新状態
- 未コミット変更なし

## 手順

### Step 1: ギャップ分析
1. README.md / SYSTEM_ARCHITECTURE.md を全文読む
2. 最終目標と現在の実装を比較し、不足している機能・品質・構造を特定する
3. 外部接続（他システムとの連携）は除外する（自分たちのシステム内に閉じたギャップのみ）
4. 結果を `docs/gap_analysis_cXX_YYYYMMDD.md` に書き出す

### Step 2: 候補列挙
1. ギャップから開発候補を列挙する（アイデア生成メンバーに委任）
2. 各候補に以下を含める:
   - 何を扱うか
   - なぜ今の構造から要請されるか
   - 既存基盤の活用度
   - 独立性（他の候補との依存関係）
3. 結果を `docs/candidates_cXX.md` に書き出す

### Step 3: 討論
1. 候補一覧を討論メンバーに検証させる
2. 各候補に「推奨」「条件付き推奨」「要再検討」の判断を出す
3. Tier分け（優先順位）を提案させる
4. 結果を `docs/discussion_cXX_YYYYMMDD.md` に書き出す

### Step 4: 優先順位決定
1. 討論結果を読み、リーダーが最終的な実装順序を決定する
2. Tier 1（即着手）/ Tier 2（次回以降）/ 保留 に分類する
3. Tier 1の各候補について正規フロー（設計→計画→解析→実装→解析→レビュー）を回す

## 注意事項
- ギャップ分析で「全部やらなきゃ」と焦らない。Tierで分けて1つずつ
- 討論で「要再検討」になった候補は無理に進めない
- 参考実装との比較分析が直近であれば、その結果も参照する
