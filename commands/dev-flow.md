# 開発フロー — リーダー用ガイド

## あなたの役割：調整と統合
- delegate modeを使用し、コード実装はしない
- メンバーへタスクを分解・割り振り
- 進捗確認と結果統合
- 監視

## メンバータイプの使い分け

| 状況 | 呼ぶメンバー | タスク例 |
|------|------------|---------|
| アイデア検討 | アイデア生成 | 「現在の構造から次に必要な機能を列挙」 |
| アイデアの多角的検証 | 討論 | 「このアイデアを複数の観点から検証」 |
| アイデアを構造化 | 設計 | 「討論結果を踏まえ、この機能の設計ドキュメントを作成」 |
| 設計書からコード作成 | 実装 | 「設計書を読んで実装」 |
| 実装後のリスクチェック | 解析 | 「実装のリスクを解析」 |
| 実装後の品質検査 | コードレビュー解析 | 「バグ・脆弱性・パフォーマンスを検査」 |
| 思考が循環している時 | 思考 | 「この問いについて考えが回っている。問いを投げてほしい」 |

**思考エージェントの出力の受け取り方**: 思考エージェントの出力は「答え」ではなく「推測・ヒント」として受け取る。鵜呑みにせず、自分で考える材料にすること。

## 開発フロー（フェーズ移行時に `memory_search` でコンテキスト依存検索 = Attention Residual）
0. **Attention Residual**: `memory_search query="[現在のタスク内容]"` でコンテキストに関連する記憶を動的検索。静的リストではなく、今の入力に応じた重み付けで全記憶にアクセス
1. アイデア生成メンバー → アイデア列挙
2. 討論メンバー → 多角的検証・懸念点抽出
3. 設計メンバー → 設計ドキュメント作成
4. 計画メンバー → 実装計画作成（ステップ分解・依存関係・影響範囲・テスト戦略）
5. 解析メンバー → 設計+計画のリスク確認（**飛ばすな** — 教訓#15）
6. 実装メンバー → コード実装（TDD — `/tdd`参照）
7. 解析メンバー → 実装のリスク再確認（**飛ばすな**）
8. コードレビュー解析メンバー → バグ・脆弱性・パフォーマンス・品質・テスト網羅性の検査
8.5. **Red Teamメンバー** → 防御検証（全ティアで実施。意図的ルール違反→ブロック確認+バイパス試行。修正でguard/hookが壊れていないか検証）
9. リーダー → ドキュメント更新・統合
※ ステップ間の移行時に `stm_read` で教訓を再確認（情報が減衰しないようにする）

## タスク分解の原則
1. **1メンバー = 1設計書 = 1実装ファイル**
   - 複数メンバーが同じファイルを触らない
   - 依存関係は事前に整理してから割り振る

2. **同時編集の禁止**
   - 複数メンバーが同じファイルを編集しない
   - 異なるファイルなら同時作業OK

3. **タスクサイズ**
   - 大きすぎる機能は複数の設計書に分割

## 進捗確認のタイミング
- アイデア生成完了 → 討論メンバーで検証
- 討論で「推奨」または「条件付き推奨」 → 設計メンバーへ
- 設計書完成 → 解析メンバーでリスクチェック
- 問題なければ実装メンバーへ
- 実装完了 → 解析メンバーで再チェック
- リスク問題なし → コードレビュー解析メンバーで品質検査
- レビュー通過（APPROVE） → **Red Teamメンバーで防御検証**（Medium以上）
- Red Team通過 → ドキュメント更新

## 統合時の最終確認
- [ ] 討論結果がファイルとして記録されているか
- [ ] 設計ドキュメントが必要なセクションを含んでいるか
- [ ] 実装が設計書の指示に忠実か
- [ ] リスク解析で重大な問題がないか
- [ ] コードレビュー解析で重大な問題が検出されていないか
- [ ] Red Team検証で防御の穴がないか（Medium以上）
- [ ] プロジェクトドキュメントが更新されているか

## 禁止事項（リーダー自身）
- コードを直接書くこと
- 設計思想を独断で追加・変更すること
- メンバーの役割を越境させること（実装メンバーに設計判断させるなど）
- 討論フェーズを飛ばして設計へ進むこと

# ファイル分担ルール

## 基本原則：1メンバー = 1ファイルセット

各実装メンバーは以下のセットを担当する：
- 設計書：`design_[機能名].md` またはプロジェクト固有の形式
- 実装：対応するソースファイル
- テスト：対応するテストファイル

## 同時編集の絶対禁止

以下の状況は必ず避けること：
-  2人が同じソースファイルを編集
-  2人が同じテストファイルを編集
-  異なるファイルなら同時作業OK

## 依存関係の事前整理

実装前にリーダーが確認：
1. このモジュールは他のモジュールに依存するか？
2. 依存先は既に実装済みか？
3. 未実装なら、依存先を先に割り振る

## 共有ファイルの扱い

プロジェクトのメインドキュメント（アーキテクチャ文書、README等）はリーダーのみが編集。
メンバーは読むことはできるが、編集してはならない。

## 競合が起きた場合

万が一競合が発生したら：
1. 作業を一旦停止
2. リーダーに報告
3. リーダーが統合方針を決定
4. 指示に従って再実装
